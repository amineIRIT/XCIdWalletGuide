Cross-Cloud Identity Management Wallet (XCId)

Step by step guide on Cross-Cloud Authentication via OpenId Connect

The XCId wallet is developed by the SIERA team from IRIT Lab and Toulouse University, France.
Mohamed Amine Ben Haj Salah, Romain Laborde, Daniele Canavese, Abdelmalek Benzekri, Mohamed Ali Kandi, and Afonso Ferreira

Aim/Goal of the project

Cross-cloud is an emerging strategy to run applications seamlessly across different Cloud Service Providers. Even though there are significant benefits such as scalability, compliance or vendor lock-in avoidance, aspects of security interoperability — particularly in the realm of identity management — remains a significant challenge. present a novel solution, called XCId, that addresses identity interoperability in cross-cloud environment without relying on intermediaries or third-party brokers. Our approach follows the Self-Sovereign Identity principles to ensure a user-centric identity solution and is built upon existing and widely deployed protocols, allowing interoperability with current Cloud Service Providers technologies. The framework was tested on Amazon Web Service, Google Cloud platform and OwnCloud. The goal of this step by step guide is to establish a complete solution capable to authenticate users and workfloads to the Cloud Service Providers via a wallet that the admin/owner have the full rights and control on. To test this solution, a python script will try to download a file from each CSP by authenticating via the OIDC tokens issued by the XCId wallet.

Identity Management Model

Overview of the identity management architecture and flow across cloud services and blockchain technologies. PKCE flow was used for the TRS API authentication and Token Exchange Standard flow for TRS to impersonate users(apps) to acquire valid access tokens for the CSPs.

Wallet Setup

a. Identity Provider - Keycloak

Purpose : the identity provider is an important component of our XCId wallet. It is responsible for issuing OIDC tokens and for the authentication process to the wallet itself by protecting the TRS API.
Create a new realm : we'll call it the sieraMultiCloudOidc realm. This realm, contains one public OIDC client for the API (X-Cloud TRS) and one confidential OIDC client (per cloud service provider). To enable the standard token exchange, the OIDC clients (csp client) should be confidential to limit the exposure of theses clients, thus securing the tokens exchange
Create new users : one per wokflow/app wanting to access the wallet. Add the cspList attribute to the user with values like csp1OidcClientId, csp2OidcClientId, etc.
Create Client scopes : one for the cspList attribute and one the CSPs wallet audiences. These scopes will ensure that the OIDC tokens include the necessary attributes for the cross-cloud authentication process
Important info about token exchange : It allows a client to exchange an access token issued to one client (e.g., the TRS public client) for a new token issued to another called a target client (e.g., a CSP client), provided the original token's audience includes the target client's ID. This is a security feature that ensures that the token is only used by the intended client and not intercepted by other clients.

After the first login, The app's access token should include the different CSPs audiences and the cspList attribute :

Cloud Service Providers Setup

This sections presents the needed steps to configure three different cloud service providers to work with our cross-cloud identity management system. Each provider has its own authentication but we'll integrate them all with our OIDC-based identity system. This integration enables seamless access to resources across different clouds while maintaining each provider's security requirements. We'll set up AWS, Google Cloud Platform, and OwnCloud, each with their specific configurations for OIDC authentication and resource access.

The setup process for each provider follows a similar pattern:

  • Configure storage resources (buckets/containers)
  • Set up OIDC authentication integration
  • Configure access policies and permissions (roles)
  • Test the integration with our identity system


a. Amazon Web Services

To set up the AWS infrastructure for cross-cloud identity management, follow these steps:

  1. Create an S3 bucket to store your files through the AWS S3 Console or using the AWS CLI. Configure appropriate bucket policies.
  2. Set up an OIDC Identity Provider in AWS IAM to allow your application to authenticate users through your Keycloak server. Follow the AWS documentation for adding an OIDC provider and configure it with your Keycloak server's OIDC endpoint and client credentials.
  3. Configure Amazon Cognito to handle the identity federation:
    • Create a new Identity Pool in the Amazon Cognito Console
    • Configure the Identity Pool to use your OIDC provider for authentication
    • Create an IAM role for authenticated users that will be assumed when users authenticate through Cognito. This could also be done in an advanced way by creating multiple roles and a policy dictating the role to lend based on the attributes in the OIDC token.
    • Create and attach an IAM policy to this role that grants the necessary permissions to access your S3 bucket (including operations like s3:GetObject, s3:PutObject, and s3:ListBucket)

For detailed step-by-step instructions, refer to these AWS documentation pages:

  • Creating an S3 Bucket
  • Adding an OIDC Identity Provider
  • Creating an Identity Pool
  • Creating IAM Roles
  • Creating IAM Policies

This setup will allow your application to authenticate users through Keycloak and then use AWS Cognito to obtain temporary AWS credentials for accessing the S3 bucket securely.

b. Google Cloud Platform

To set up the Google Cloud Platform infrastructure for cross-cloud identity management, follow these steps:

  1. Create a Cloud Storage bucket to store your files:
    • Go to the Cloud Storage Browser
    • Click "Create Bucket" and follow the setup wizard
    • Choose a globally unique name for your bucket
    • Select the appropriate storage class and location
    • Configure access control settings:
      • Set uniform bucket-level access
      • Set up appropriate IAM policies
  2. Add an Identity Provider in Google Cloud IAM:
    • Navigate to the Workload Identity Federation page
    • Create a new Workload Identity Pool
    • Configure the OIDC provider with your Keycloak server's OIDC endpoint and client credentials
    • Set up attribute mapping to map Keycloak claims to Google Cloud attributes
  3. Create a Service Account that will be used by authenticated users:
    • Go to the Service Accounts page
    • Create a new service account with appropriate permissions
    • Grant the service account access to Google Cloud Storage (roles/storage.objectViewer, roles/storage.objectCreator)
    • Download the service account key. This key is a json file containing the filepath of the OIDC token. This is a necessary step to be able to use the service account in the workload. The "key" file stays always the same, but the value of the OIDC access token changes after each login (this sould be done by the app itself after receiving the OIDC tokens from TRS).
  4. Configure Workload Identity Federation:
    • Create a Workload Identity Pool in the Workload Identity Federation console
    • Configure the pool to use your OIDC provider
    • Set up attribute conditions to control which users can access which resources
    • Create a service account impersonation configuration

For detailed step-by-step instructions, refer to these Google Cloud documentation pages:

  • Creating Cloud Storage Buckets
  • Workload Identity Federation Overview
  • Configuring Workload Identity Federation
  • Creating and Managing Service Accounts
  • Cloud Storage IAM Permissions

This setup enables your application to authenticate users through Keycloak and then use Workload Identity Federation to obtain temporary Google Cloud credentials for accessing Cloud Storage securely. The service account will be impersonated by authenticated users based on the attributes in their OIDC tokens.

c. OwnCloud & OIDC App

To set up OwnCloud with OIDC authentication integration for cross-cloud identity management, follow these steps:

  1. Install the OIDC App on OwnCloud through the OwnCloud Marketplace.
  2. Beware of this OIDC php bug, run this command to ensure the PKCE flow (bug explained here): 
    occ config:system:set 'http.cookie.samesite' --value Lax
  3. Configure the OIDC App using the OwnCloud CLI (occ):
    • Access your OwnCloud server's command line
    • Use the following configuration command: 
      occ config:app:set openidconnect openid-connect 
    --value='{"provider-url":"http://XXXXXX/realms/sieramulticloudoidc",
    "client-id":"OwnCloudClient",
    "client-secret":"OwnCloudClientSecret",
    "loginButtonName":"Login via SIERA OpenId Connect", 
    "search-attribute":"preferred_username", 
    "auto-provision":{"enabled":false}, 
    "mode":"userid"}'
    • Configuration parameters explained:
      • provider-url: Your Keycloak server's OIDC endpoint
      • client-id: The OIDC client ID registered in Keycloak
      • client-secret: The client secret from Keycloak
      • search-attribute: The attribute used to identify users
      • auto-provision: Whether to automatically create users
      • mode: The user identification mode
  4. Configure OwnCloud Storage:
    • OwnCloud acts like a private cloud where every user own a dedicated storage space. Therefore, the apps will login as a user (a human) and not as a workfload/workflow like AWS and GCP.
    • Set up appropriate storage quotas for users
    • Configure file sharing permissions
    • Set up group policies if needed

For detailed step-by-step instructions, refer to these OwnCloud documentation pages:

  • OwnCloud OAuth2/OIDC Authentication
  • OwnCloud Command Line Tool
  • File Sharing Configuration

This setup enables your application to authenticate users through Keycloak and access OwnCloud storage securely.

Application DID Implementation

import indy_vdr.ledger as indyledger
from indy_vdr import Pool, open_pool

#Function to get the TRS link from the blockchain
async def getTRSlink(walletDID):
    # Constants
    GENESIS_FILE_PATH = "XXX"
    pool = Pool

    # Step 1: Open the pool
    pool = await open_pool(transactions_path=GENESIS_FILE_PATH)

    # Step 2: Query the attribute (GET_ATTRIB)
    get_attrib_request = indyledger.build_get_attrib_request(None, walletDID, "TRSlink", None, None)
    attrib_query_response = await pool.submit_request(get_attrib_request)
    
    # Step 3: Extract the URL from the response
    goal=attrib_query_response['data']
    # Find the start and end positions of the URL
    start_pos = goal.find('"url":"') + len('"url":"')
    end_pos = goal.find('"', start_pos)
            
    # Step 4: Close the pool
    pool.close()

    return goal[start_pos:end_pos]

This Python implementation demonstrates how to retrieve the TRS link from the blockchain using the Hyperledger Indy VDR library.

Cross-Cloud Application/workload Example

This section presents an example of what a cross-cloud app could be. The code shown below is from an example app, where it authenticates with the IdP, gets multiple OIDC tokens and use them to download a specific file from the three CSPs.

Function Reference

Function Parameters Description
getTRSlink() walletDID: str Retrieves the TRS (Token Retrieval Service) URL from the blockchain. Connects to the Hyperledger Indy pool, queries the wallet's TRSlink attribute, and returns the URL.
extract_access_token() json_str: str, key: str Extracts an OIDC token from a JSON response for a specific CSP. Parses the JSON string and returns the access token for the specified key.
extract_field_token() access_token: str, field_name: str Extracts a specific field from a JWT token payload. Decodes the base64 payload and returns the requested field value.
get_trsToken_pkce() keycloak_url: str, client_id: str, redirect_uri: str Gets OIDC token from IDP using PKCE flow. Implements the OAuth2 PKCE authentication flow with Keycloak, including code verifier/challenge generation and token exchange.
get_tokens_trs() keycloak_url: str, csplist: str Gets tokens from the Token Retrieval Service for all CSPs. First gets an IDP token, then uses it to request CSP-specific tokens.
download_file_oc() access_token: str, file_url_oc: str, file_destination: str Downloads a file from OwnCloud using OIDC token. Makes an authenticated request to OwnCloud's WebDAV API and saves the file locally.
download_file_gcp() token: str, bucket_name: str, source_blob_name: str, destination_file_name: str Downloads a file from Google Cloud Storage using OIDC token. Uses Google Cloud Storage client library with token-based authentication.
download_file_aws() JWToken: str Downloads a file from AWS S3 using OIDC token and Cognito. Exchanges OIDC token for AWS credentials via Cognito Identity Pool and downloads the file.

Function Call Flow:

  1. Main execution starts with get_tokens_from_trs()
  2. get_tokens_from_trs() calls get_trsAgent_token_from_idp() for authentication
  3. For each CSP in the response:
    • Extract token fields using extract_field_from_token()
    • Call appropriate download function based on CSP type:
      • download_file_oc() for OwnCloud
      • download_file_gcp() for Google Cloud
      • download_file_aws() for AWS
